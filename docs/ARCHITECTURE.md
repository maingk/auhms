# **2\.  Confirmed Architectural Decisions**

The following decisions have been made and should not be revisited by Claude unless explicitly asked. Do not suggest alternatives to these choices.

## **2.1  Identity & Authentication**

✓ Authentication via Azure Active Directory (Microsoft Entra ID) SSO using NextAuth.js — students and staff log in with existing university Microsoft credentials. No separate user account system will be built.

✓ Role-based access control (RBAC) is required from day one. Do not suggest adding it later. Roles include at minimum: Housing Admin, Housing Staff, Student, IT Admin, Read-Only Reporter.

## **2.2  Microsoft 365 Integration**

✓ The application is part of a Microsoft 365 ecosystem. Email notifications will use the Microsoft Graph API, not a third-party email service.

✓ Teams notifications will be used for internal staff alerts, consistent with existing Power Automate workflows already in use at the institution.

✓ SharePoint may be used for document storage (lease agreements, inspection forms). Graph API is the integration path.

## **2.3  ERP Integration**

✓ Colleague (Ellucian) is the institution's ERP. Student identity and enrollment data originates in Colleague and must be treated as the system of record for student information.

✓ Billing charges will be written back to Colleague rather than processed within AUHMS. AUHMS is not a payment processor.

✓ A nightly data sync (not real-time API) is the assumed integration pattern for Colleague until technical investigation proves otherwise. Do not assume a live API is available.

## **2.4  Data & Privacy**

✓ Student data is subject to FERPA. All access to student records must be logged. Do not suggest features that would expose student PII without role-gating.

✓ Soft deletes only — no hard deletion of student or assignment records. Use a deleted\_at timestamp pattern.

✓ A parallel staging environment will be maintained throughout development. Production student data will never be used in development or testing.

# **4\.  Database Conventions**

These conventions apply to ALL database work. Claude must follow these standards in every schema, migration, and query it generates.

| Engine | PostgreSQL |
| :---- | :---- |
| **Naming** | All tables and columns use snake\_case |
| **Primary keys** | UUID type, column named id, generated by default as gen\_random\_uuid() |
| **Foreign keys** | Named using pattern: referenced\_table\_singular\_id  (e.g., student\_id, room\_id) |
| **Timestamps** | All tables include created\_at and updated\_at (TIMESTAMPTZ, NOT NULL, default NOW()) |
| **Soft deletes** | deleted\_at TIMESTAMPTZ NULL — NULL means active, non-NULL means deleted. Never hard delete student or assignment records. |
| **Booleans** | Use TRUE/FALSE — never 0/1 or Y/N strings |
| **Enums** | Use PostgreSQL native ENUM types for fixed-value fields (e.g., room type, assignment status) |
| **Migrations** | *⚠ Migration tool not yet decided (Prisma Migrate, Flyway, or custom). TBD.* |

# **5\.  Core Data Model (Entity Map)**

This is the high-level entity relationship map, not the full schema. Use this to understand how data connects before writing any queries or schema changes. Full schema will be maintained separately as it grows.

## **5.1  Key Entities**

| buildings | Physical buildings on campus. Parent of rooms. |
| :---- | :---- |
| **rooms** | Individual rooms/units within a building. Contains capacity, type, amenities, status. |
| **students** | Sourced from Colleague via nightly sync. Not created manually in AUHMS. UUID maps to Colleague student ID. |
| **housing\_applications** | A student's application for housing for a given term. One per student per term. |
| **room\_assignments** | The active link between a student and a specific room for a term. One student, one room, one term. |
| **roommate\_preferences** | Student-expressed roommate preferences stored against an application. |
| **maintenance\_requests** | Submitted by students or staff against a specific room. Has status lifecycle. |
| **leases** | Formal lease record tied to an assignment. Document stored in SharePoint (link stored here). |
| **billing\_charges** | Charges generated by AUHMS to be exported/synced to Colleague. Not processed internally. |
| **audit\_log** | Immutable log of all access to and changes of student records. Required for FERPA compliance. |

## **5.2  Key Relationships**

* A building has many rooms

* A room belongs to one building

* A student can have one housing\_application per term

* An application can result in one room\_assignment

* A room\_assignment links one student to one room for one term

* A room can have many room\_assignments across terms, but only one active assignment at a time

* A maintenance\_request belongs to a room (not a student, since rooms persist between tenants)

* billing\_charges are generated from room\_assignments and exported to Colleague — never processed internally

# **6\.  Integration Notes**

## **6.1  Colleague ERP**

* Colleague is the system of record for student identity, enrollment status, and billing

* Student sync is assumed to be a nightly batch process — the exact mechanism (SFTP file drop, REST API, database view) is not yet confirmed and must be investigated

* Do not design features that assume real-time Colleague data availability

* Billing charges are written to Colleague; AUHMS should produce an exportable charge file in the format Colleague expects (format TBD)

* Colleague integration is expected to be the highest-complexity, highest-risk component of the project

## **6.2  Microsoft 365 / Azure AD**

* Authentication: Azure AD via NextAuth.js (or equivalent). Users must not be able to create standalone AUHMS accounts

* Email: Microsoft Graph API — send transactional emails (application confirmations, assignment notices, maintenance updates) using the university's mail infrastructure

* Teams: Post notifications to designated housing staff channels for new applications, maintenance requests, and assignment changes

* SharePoint: Store lease documents and inspection forms. Reference by URL in the leases table. Graph API for upload/retrieval

* The existing Power Automate environment should be considered for integration triggers where appropriate, rather than building parallel notification systems

## **6.3  Incumbent System — Adirondack Solutions: The Housing Director**

AUHMS is a direct replacement for The Housing Director. Understanding this system's capabilities and data structures is important context for requirements gathering and data migration planning.

* **Incumbent system — a widely used higher education housing management platform by Adirondack Solutions:** The Housing Director

* **Expiring — this is the primary driver for building AUHMS. Confirm exact expiration date and negotiate a short-term extension if needed to provide development runway:** Contract status

* **Historical data currently in The Housing Director will need to be evaluated for migration into AUHMS. At minimum, current room inventory and active resident records should migrate. Historical assignment records are lower priority but worth preserving if export is feasible:** Data migration

* **Before finalizing requirements, conduct a structured walkthrough of every feature your housing staff currently uses in The Housing Director. Do not assume you know all workflows — staff often use features that are invisible to IT. This is a key task for your housing rep:** Feature parity

* **Adirondack typically provides data export capabilities. Early in the project, request a full data export from The Housing Director in whatever formats are available (CSV, XML, etc.) — this gives you real data to inform schema design and test migration scripts against:** Data export

* **Even after go-live, keep Adirondack contact information on file in case historical data questions arise post-migration:** Vendor relationship

## **6.4  Data Migration Strategy**

*⚠ DECISION NEEDED: Full migration scope not yet defined. The items below are recommended starting points for discussion.*

* **Current room and building inventory, active student housing assignments, current lease records:** Priority 1 (must migrate)

* **Pending applications for upcoming terms, maintenance request history for open items:** Priority 2 (should migrate)

* **Historical assignment records from prior years — evaluate storage vs. value tradeoff:** Priority 3 (archive or skip)

* **Write and test migration scripts against exported Adirondack data before go-live. Run in staging first. Never migrate directly to production without a validated dry run:** Migration approach

* **Define a clear cutover date and data freeze point — a moment when The Housing Director stops being updated and AUHMS becomes the live system. Parallel running before cutover is strongly recommended for at least one housing cycle:** Cutover plan
